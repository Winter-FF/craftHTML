API Rate Limiting System

Author Team Platform
Date May 10, 2026
Status Draft
Type Technical Specification

Summary

A sliding-window rate limiting system for the public API, using Redis for distributed state. Supports per-tenant and per-endpoint limits with three pricing tiers. Target: 10K+ req/s with <5ms added latency.

Overview

Our current API has no rate limiting. This has led to several incidents where a single tenant consumed all available resources, causing degradation for other users. We need a fair, transparent, and configurable rate limiting system.

Stakeholder Requirements
  • Must support per-tenant limits
  • Must support per-endpoint limits
  • Must provide clear feedback when limits are hit
  • Must not add more than 5ms latency per request
  • Must be observable and auditable

Architecture

The rate limiter sits between the API gateway and the application layer. It uses a sliding window counter algorithm stored in Redis for distributed state management.

Components

Component Responsibility
Rate Limiter Middleware Express middleware intercepting all API requests
Redis Cluster Stores sliding window counters with TTL-based expiry
Config Service Manages per-tenant and per-endpoint limit configurations
Metrics Collector Emits rate limiting metrics to the observability stack

Data Flow

Request → API Gateway → Rate Limiter → Redis → 200 / 429

Redis Key Schema

Key Pattern
ratelimit:{tenant_id}:{endpoint}:{window_start}

// Example
ratelimit:acme-corp:/api/users:1715337600
Sliding Window Algorithm
  1. Calculate the current window timestamp (floor to window size)
  2. Calculate the weight of the previous window based on overlap
  3. weighted_count = current + (previous × overlap_percentage)
  4. If weighted_count ≥ limit, reject with 429

This provides smoother rate limiting than fixed windows while being more memory-efficient than true sliding window logs.

Response Format

When rate limited, return HTTP 429 with the following JSON body and headers:

JSON Response{
  "error": "rate_limit_exceeded",
  "message": "Rate limit exceeded. Try again in 23 seconds.",
  "retry_after": 23,
  "limit": 600,
  "remaining": 0,
  "reset_at": "2026-05-10T14:30:00Z"
}
Header Description
X-RateLimit-Limit Maximum requests per window
X-RateLimit-Remaining Requests remaining in current window
X-RateLimit-Reset Unix timestamp when the window resets
Retry-After Seconds until the client should retry

Error Handling

Failure Strategy Behavior
Redis down Fail Open Allow requests, log alert
Config service down Cache Use cached config, expire after 5 min
Counter overflow Cap Cap at 2× limit, log anomaly

Open Questions

Should we support per-second sliding windows for burst protection?

Current design only supports per-minute granularity. Per-second windows would provide tighter burst control but increase Redis write volume significantly. Needs load testing to determine if the tradeoff is worthwhile.

How do we handle rate limits for webhook deliveries?

Webhooks have a different traffic pattern (outbound, bursty, retry-heavy). They may need separate rate limit buckets or a different algorithm entirely. Needs further investigation.

Should rate limit headers be included on all responses or only on 429s?

Including headers on all responses helps clients implement proactive throttling but adds slight overhead to every response. Recommendation: include on all responses for the best developer experience.
